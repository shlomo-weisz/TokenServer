# TokenLearn Server Architecture

## Overview

The server is a Spring Boot 3 application that exposes the TokenLearn backend API.
It handles:

- authentication and JWT issuance
- user profiles, course selections, and availability
- tutor discovery
- lesson request creation and approval
- lesson lifecycle management
- token reservations, refunds, transfers, and settlement
- inbox notifications
- admin operations

The application uses SQL Server for the system of record, Flyway for schema
migrations, and ActiveMQ Artemis for asynchronous settlement processing.

## Core Runtime Flow

### 1. Authentication

`AuthController` calls `AuthService` for login, registration, password reset, and
Google sign-in.

- Password-based login loads the user from `UserDao` and validates the BCrypt hash.
- Successful login returns a JWT generated by `JwtProvider`.
- `JwtAuthenticationFilter` parses the Bearer token on later requests and stores
  the authenticated user id in the Spring Security context.
- Admin permissions are enforced in application code (controllers and services),
  not from JWT roles.

### 2. Profile and Tutor Data

`UserService` and `TutorService` aggregate data from several DAOs:

- `users`
- `courses`
- `user_courses_teacher`
- `user_courses_student`
- `availability`
- `ratings`

This layer intentionally assembles API-shaped payloads so controllers do not need
to join multiple repositories themselves.

### 3. Lesson Request Lifecycle

`LessonRequestService` owns the request workflow:

1. A student creates a lesson request.
2. A `lesson_requests` row is inserted with status `PENDING`.
3. The student's balance is reserved by moving tokens from `available_balance`
   to `locked_balance`.
4. A `RESERVATION` transaction is written to `token_transactions`.
5. A notification is created for the tutor.

When a tutor approves the request:

1. The request transitions from `PENDING` to `APPROVED`.
2. A `lessons` row is created with status `SCHEDULED`.
3. The student's tokens remain locked until the lesson is completed or cancelled.

When a request is rejected, cancelled, or expires:

1. Status changes away from `PENDING`.
2. Locked tokens are released back to the student's available balance.
3. A `REFUND` transaction is written.

### 4. Lesson Completion and Settlement

`LessonService` separates lesson completion from token settlement.

When a lesson is completed:

1. The lesson transitions from `SCHEDULED` to `COMPLETED`.
2. The related request is updated to `COMPLETED`.
3. A settlement payload is written to `jms_outbox`.

No direct tutor credit happens in that transaction. Instead:

1. `OutboxPublisher` polls `jms_outbox`.
2. `LESSON_COMPLETED` events are sent to the Artemis `settlement-queue`.
3. `SettlementConsumer` receives the payload.
4. `SettlementService` performs the actual balance move from
   `student.locked_balance` to `tutor.available_balance`.
5. A `SETTLEMENT` transaction is written to `token_transactions`.

This design keeps lesson completion durable even if the message broker or
settlement consumer is temporarily unavailable.

## Idempotency and Audit Trail

The financial flow is intentionally traceable:

- every purchase, reservation, refund, transfer, bonus, admin adjustment, and
  settlement is persisted in `token_transactions`
- settlement retries are safe because the service first checks whether a
  `SETTLEMENT` transaction already exists for the request
- the schema also enforces uniqueness for key transaction combinations and
  `message_id`

## Notifications

`NotificationService` stores inbox-style notifications in SQL Server instead of
depending on a websocket or push provider.

Current notification types:

- `LESSON_REQUEST_CREATED`
- `LESSON_REQUEST_APPROVED`
- `LESSON_REQUEST_REJECTED`
- `LESSON_CANCELLED`
- `LESSON_REMINDER`
- `LESSON_MESSAGE`

Lesson messages are stored as paired notifications: the recipient gets an unread
notification, and the sender gets a read copy so the conversation can be
reconstructed from the inbox stream.

## Scheduled Jobs

Several recurring jobs are enabled through `@EnableScheduling`:

- `LessonRequestService.expirePendingRequests()`
  expires pending requests that are too close to the requested lesson time
- `LessonService.autoCompletePastLessons()`
  completes scheduled lessons whose end time has passed
- `NotificationService.createUpcomingLessonReminders()`
  creates reminder notifications about one hour before a lesson
- `OutboxPublisher.publishNewEvents()`
  sends pending outbox rows to Artemis
- `OutboxPublisher.retryFailed()`
  requeues failed outbox rows for another publish attempt

## Persistence Notes

- Flyway migrations live under `src/main/resources/db/migration`.
- The base schema is created in `V1__Initial_Schema.sql`.
- Later migrations add course catalog columns, Unicode support, notifications,
  inbox message fields, and the `EXPIRED` lesson-request status.

## Course Catalog Import

`CourseCatalogSyncService` runs at startup when `app.course-catalog.enabled=true`.
It loads `openu_all_courses.json`, resolves a valid path, and upserts courses by
course number and names. This makes the backend resilient to catalog refreshes
without requiring manual SQL scripts.
